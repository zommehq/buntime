# Deployment

Vault is shipped as a worker app under `apps/vault` and loaded by Buntime runtime from `RUNTIME_WORKER_DIRS`.

## Packaging

Build artifacts are generated by:

```bash
bun run --filter @buntime/vault build
```

Artifacts:

- `dist/index.js`: worker entrypoint bundle.
- `dist/client/*`: SPA assets.
- `dist/migrations/*`: SQL migrations copied from server source.

## Worker Manifest

`apps/vault/manifest.yaml` controls runtime behavior:

- `entrypoint: dist/index.js`
- `injectBase: true`
- `visibility: protected`
- database provider: `resource-tenant`
- migration folder: `migrations`

## Runtime Route Surface

For app name `vault`:

- SPA base: `/vault`
- API in app context: `/vault/api`
- Vault API: `/vault/api/vault/*`
- Health: `/vault/health`
- OpenAPI: `/vault/openapi.json`
- Scalar docs: `/vault/docs`

## Required Environment

- `VAULT_MASTER_KEY` for any endpoint that encrypts/decrypts secret values.
- DB connectivity through either:
  - runtime-injected `x-database-url`, or
  - `DATABASE_URL`, or
  - `PGLITE_PATH` (dev-only).

## Production Checklist

1. Build and ship `dist/` contents.
2. Ensure migrations are included in deploy artifact.
3. Set `VAULT_MASTER_KEY` to a secure 32-byte base64 key.
4. Ensure PostgreSQL connectivity and tenant routing are configured.
5. Validate health and docs endpoints:
   - `/vault/health`
   - `/vault/docs`
   - `/vault/openapi.json`
6. Validate secret operations with `VAULT_MASTER_KEY` present:
   - `/vault/api/vault/:id/reveal`
   - `/vault/api/vault/resolve`
   - `/vault/api/vault/secrets/expiring`

## Notes

- No compatibility redirect is configured for legacy `/parameters` routes.
- Current database schema namespace remains `parameters` for migration/data continuity.
- `injectBase: true` is required so SPA and relative `api` client work correctly under `/vault`.
