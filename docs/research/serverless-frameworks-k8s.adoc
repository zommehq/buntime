== Contexto

Análise de viabilidade para servir aplicações multi-tenant usando frameworks serverless no Kubernetes, como alternativa ou evolução à arquitetura atual do Buntime.

O Buntime é um runtime multi-tenant baseado em Bun + Hono com pool de workers, sistema de plugins com hooks (`onRequest`, `onResponse`, `provides`/`getPlugin`), gateway, authn/authz, e gerenciamento de lifecycle.

== Capacidades-chave a mapear

[cols="1,3"]
|===
| Capacidade | Descrição

| Gateway/Routing
| Roteamento de requests para apps por path/host, traffic splitting, canary

| Auth Pipeline
| Interceptação de requests (authn -> authz -> CSRF), equivalente ao `onRequest` chain

| Plugin/Extensibilidade
| Adicionar capacidades (database, keyval, metrics, logs) sem tocar no core

| Inter-service
| Um serviço/plugin chamar outro (equivalente a `getPlugin<T>()`)

| Multi-tenancy
| Isolamento por tenant (database, config, rotas)

| Lifecycle
| Scale-to-zero, health checks, deploy/rollback

| Observabilidade
| Metrics, logs, tracing integrados
|===

Todos os frameworks são open source com licença *Apache 2.0*, permitindo uso comercial, modificação e distribuição sem royalties ou custos de licença. A única obrigação é manter o aviso de licença e copyright nos arquivos fonte. Nenhum possui modelo "open core" com features pagas no runtime.

NOTE: Existem serviços gerenciados pagos (Fermyon Wasm Functions/Akamai para Spin, Google Cloud Run para Knative, Azure Container Apps para Dapr+KEDA), mas o software em si é totalmente livre para self-host.

=== SpinKube (Fermyon Spin)

*O que é:* Framework para microserviços WebAssembly. Cada app é um componente Wasm executado pelo runtime Wasmtime. SpinKube é o operador Kubernetes que gerencia SpinApps como CRDs.

*Pontos fortes:*

* Cold start sub-milissegundo (<1ms)
* Isolamento por sandbox Wasm (capability-based)
* Densidade extrema: milhares de instâncias por nó
* OTel Factor nativo para observabilidade
* Component Dependencies (Spin 3.0+) para composição via WIT

*Limitações:*

* Sem middleware pipeline -- composição de auth é estática (build-time via `wasm-tools compose`)
* Sem WebSocket (request/response puro)
* Sem estado em memória entre requests (cada request = instância nova)
* JS/TS via ComponentizeJS tem overhead e gaps
* Sem sistema de plugins dinâmico
* Precisa de gateway externo (Envoy, Istio, etc.)

=== Knative Serving

*O que é:* Plataforma serverless para containers no Kubernetes. Provê scale-to-zero, traffic management (revisions, canary, blue-green), e auto-scaling baseado em concorrência.

*Pontos fortes:*

* CNCF Graduated -- máxima maturidade
* Scale-to-zero nativo com activator
* Traffic splitting, revisions, rollback instantâneo
* Integração com Istio para auth (RequestAuthentication + AuthorizationPolicy)
* Base do Google Cloud Run

*Limitações:*

* Cold start de containers: 1-5s típico (até ~17s worst case com JVM)
* *Multi-tenancy é o ponto mais fraco* -- activator/autoscaler são singletons cluster-wide (issue #12533 em Icebox)
* Sem sistema de plugins nativo
* Auth depende inteiramente do Istio (RequestAuthentication + AuthorizationPolicy)
* Istio consome ~1-2GB+ de RAM

=== Dapr (Distributed Application Runtime)

*O que é:* Runtime baseado em sidecar que provê building blocks para aplicações distribuídas: service invocation, state management, pub/sub, bindings, middleware pipeline, workflows. Language-agnostic.

*Pontos fortes:*

* CNCF Graduated, 40k+ organizações
* *Middleware pipeline declarativo* (`httpPipeline`) -- chain de middlewares YAML: bearer -> OPA -> ratelimit -> WASM custom. Modifica request/response. *Mais próximo do `onRequest` chain do Buntime*
* *Pluggable Components* -- custom gRPC servers em qualquer linguagem. Hot reload via Operator
* Service invocation com mTLS automático, service discovery, retry, circuit breaker
* *Multi-tenancy forte*: namespace isolation + component scoping + ACLs per-app/per-verb/per-path
* State stores: Redis, PostgreSQL, MongoDB, DynamoDB, CosmosDB, SQLite, e 30+ opções
* OTel nativo com W3C Trace Context propagation automática
* Key prefix por tenant no state store
* Resiliency CRD (retries, circuit breakers, timeouts)

*Limitações:*

* Não é gateway -- precisa de Envoy/Istio/Kong na frente
* Sidecar overhead (~4MB RAM por pod, ~58MB binário)
* Scale-to-zero via KEDA é mais lento que Knative (~5-15s)
* Middleware WASM custom usa http-wasm ABI (não é o mesmo que Spin/wasmCloud Wasm)

=== wasmCloud

*O que é:* Plataforma distribuída baseada em WebAssembly com modelo de capability providers. Componentes (Wasm) declaram necessidades via WIT, providers satisfazem essas necessidades. Comunicação via NATS lattice.

*Pontos fortes:*

* *Capability Providers* -- modelo mais próximo do `provides()`/`getPlugin()` do Buntime. Componentes declaram imports WIT, providers satisfazem
* Lattice distribuído via NATS -- multi-host, multi-região sem service mesh externo
* Multi-lattice para isolamento de tenants
* Policy service com OPA integrado
* Assinatura criptográfica de artefatos (JWT com claims de capabilities)
* Hot-swap de providers (v1, parcial no v2)
* wash dev com detecção automática de providers a partir de WIT

*Limitações:*

* Transição v1->v2 com mudanças arquiteturais significativas
* Providers devem ser escritos em Rust ou Go (não TypeScript)
* NATS como dependência hard de infraestrutura
* Multi-tenancy básica (sem quotas, sem billing isolation)
* Ecossistema menor e menos testado em produção que Dapr/Knative
* Complexidade operacional maior (NATS cluster + JetStream + wadm + operator)

== Mapeamento por capacidade

=== Gateway/Routing

[cols="1,2,1"]
|===
| Framework | Como funciona | Maturidade

| SpinKube
| SpinApp CRD gera K8s Service. Precisa de Ingress/Gateway API externo
| Básico

| Knative
| Ingress gateway nativo (Kourier/Istio/Contour). Revisions, traffic splitting, canary
| *Excelente*

| Dapr
| Não é gateway -- precisa de Envoy/Istio/Kong. Cuida do service-to-service interno
| Complementar

| wasmCloud
| HTTP capability provider + lattice routing. Gateway externo necessário
| Básico
|===

[cols="1,2,1"]
|===
| Framework | Como funciona | Maturidade

| SpinKube
| Build-time composition (`wasm-tools compose` auth.wasm + app.wasm). Estático
| Limitado

| Knative + Istio
| `RequestAuthentication` (JWT) + `AuthorizationPolicy` (RBAC) declarativos no mesh
| Produção

| Dapr
| `httpPipeline` declarativo -- chain de middlewares YAML: bearer -> OPA -> ratelimit -> WASM custom
| *Excelente*

| wasmCloud
| Policy service com OPA. Custom auth via capability provider
| Funcional
|===

|===

== Arquiteturas propostas

=== Opção A: Dapr + KEDA + Envoy Gateway (mais pragmático)

[source]
----
Internet -> [Envoy Gateway] -> [Dapr Sidecar <- httpPipeline: auth/opa/rate] -> App
                                     |  invoke (mTLS)
                               [Dapr Sidecar] -> App B
                                     |  state/pubsub/binding
                               [Components: Redis, PG, Kafka...]
----

* *Gateway:* Envoy Gateway (K8s Gateway API)
* *Auth:* Dapr `httpPipeline` com bearer/OPA/custom WASM middleware
* *Plugins:* Dapr Components + Pluggable Components custom (gRPC)
* *Multi-tenancy:* Namespace + component scoping + ACLs
* *Scale-to-zero:* KEDA ScaledObjects por app
* *Estado:* Dapr state store (Redis/PG) com key prefix por tenant
* *Apps:* Containers normais (Bun, Node, Go, qualquer linguagem)

*Prós:* CNCF Graduated, 40k+ orgs, middleware pipeline declarativo, hot reload, qualquer linguagem, menor esforço de migração.

*Contras:* Cold start do KEDA mais lento que Knative (~5-15s). Sidecar overhead (~4MB RAM por pod).

=== Opção B: Knative + Dapr + Istio (mais completo)

[source]
----
Internet -> [Istio Ingress + RequestAuth] -> [Knative Activator] -> [Pod: App + Dapr Sidecar]
                                                                          |
                                                                    [Dapr Components]
----

* *Gateway:* Istio Ingress Gateway + Knative routing (revisions, traffic split, canary)
* *Auth:* Istio RequestAuthentication (JWT) + Dapr OPA middleware
* *Scale-to-zero:* Knative KPA (1-2s cold start)
* *Plugins:* Dapr Components + Dapr service invocation
* *Multi-tenancy:* Istio mTLS + AuthorizationPolicy per namespace

*Prós:* Melhor scale-to-zero, traffic management maduro (canary, blue-green), auth no mesh level.

*Contras:* Complexidade operacional alta (3 control planes: Knative + Dapr + Istio). Istio consome ~1-2GB+ de RAM.

=== Opção C: SpinKube + Dapr (mais inovador)

[source]
----
Internet -> [Envoy Gateway] -> [SpinApp Pod: Wasm runtime + Dapr Sidecar]
                                     |  Dapr invoke/state/pubsub
                               [Dapr Components]
----

* *Apps:* Wasm components (JS/TS via ComponentizeJS, Rust, Go, Python)
* *Gateway:* Envoy Gateway
* *Auth:* Dapr httpPipeline (sidecar intercepta antes do Spin) ou build-time composition
* *Multi-tenancy:* Wasm sandbox (mais forte) + Dapr namespace isolation
* *Density:* Milhares de apps por nó
* *Cold start:* <1ms

*Prós:* Melhor isolamento (Wasm), melhor densidade, cold start imbatível. Dapr fornece building blocks que o Spin não tem.

*Contras:* Integração SpinKube + Dapr sidecar não é documentada/testada. JS/TS no Wasm tem overhead. Sem WebSocket. Risco de pioneirismo.

== Recomendação

[cols="1,1"]
|===
| Se o objetivo é... | Escolha

| Menor risco, maior maturidade
| *Opção A (Dapr + KEDA + Envoy Gateway)* -- tudo CNCF Graduated, pipeline de middleware nativo, qualquer linguagem

| Melhor scale-to-zero
| *Opção B (Knative + Dapr + Istio)* -- aceite a complexidade operacional

| Máximo isolamento e densidade
| *Opção C (SpinKube + Dapr)* -- aposta tecnológica com melhor perfil de segurança e performance

| Mais fiel ao modelo de plugins
| *wasmCloud* -- capability providers são o análogo mais direto, mas v1->v2 transição e Rust/Go obrigatório são barreiras
|===

Para o contexto atual (TypeScript/Bun, multi-tenant, plugins existentes), a *Opção A* minimiza reescrita: apps continuam como containers (podem manter Bun), e o Dapr assume responsabilidades que hoje são dos plugins do Buntime (state, auth middleware, service discovery, observability). A migração seria incremental -- mover um plugin por vez para um Dapr Component.
