# QMD — configuration for this wiki

> Concrete commands to index this wiki with [QMD](https://github.com/tobi/qmd) (local search: BM25 + vector + LLM reranking). The generic wiki-skills guide (installation, multilingual embedding model, MCP daemon) is at [`essential-skills/docs/wiki/qmd-setup.md`](https://github.com/djalmajr/essential-skills/blob/main/docs/wiki/qmd-setup.md). This file only contains what is specific to the Buntime wiki.

> **Deliberately restricted scope**: we only index the `wiki/` directory of this repo. Source code, plugins, charts, and original docs that still exist outside `wiki/` are **not** indexed — the wiki is the canonical source and any information outside it is considered a pending migration.

## Convention: named index `buntime`

All commands below pass `--index buntime`. This isolates the Buntime wiki's SQLite database at `~/.cache/qmd/buntime.sqlite`, separate from other projects you may index (`--index otherproject`, `--index personal`, etc.). Each `--index` is an independent search universe — no crosstalk.

The configuration `~/.config/qmd/buntime.yml` (automatically generated by the commands below) records the collections + contexts for this index.

## Prerequisites — patched local qmd + multilingual embedding model

> qmd 2.1.0 as published on npm has three issues that break the desired setup for this wiki:
>
> 1. **[Issue #552](https://github.com/tobi/qmd/issues/552)** — `qmd embed` ignores the `QMD_EMBED_MODEL` env var and the `models.embed` YAML field, always using `embeddinggemma-300M` (optimized for English). For multilingual content we want `Qwen3-Embedding-0.6B` (multilingual, MTEB top).
> 2. **MCP server opens the wrong index when invoked with `--index <name>`** — `startMcpServer()` in `src/mcp/server.ts` calls `getDefaultDbPath()` without passing the index name, always pointing to `~/.cache/qmd/index.sqlite` (empty default) even when the CLI was started with `--index buntime`.
> 3. **Semantic queries reject hyphenated words** — the validation for `vec:`/`hyde:` treats any `-\w` as a negation token, so natural terms such as `built-in`, `multi-agent`, `gpt-4`, and `client-side` fail before search. This is especially harmful for this wiki because `built-in` vs `uploaded` apps/plugins is a canonical runtime distinction.

Solution: clone the repo, apply local patches, install with `bun link`, and use that binary instead of the one published on npm.

### 1. Clone and apply patch

```sh
mkdir -p ~/Developer/github && cd ~/Developer/github
git clone https://github.com/tobi/qmd.git
cd qmd
```

**Patch 1 — respect `QMD_EMBED_MODEL` in the `embed` command** (in `src/cli/qmd.ts`):

- Add 3 helpers (`resolveEmbedModel`, `resolveRerankModel`, `resolveGenerateModel`) that read in order: `config.models.{embed,rerank,generate}` from YAML → env var (`QMD_{EMBED,RERANK,GENERATE}_MODEL`) → default.
- Replace the 3 hardcoded uses of `DEFAULT_EMBED_MODEL_URI` (status display, `embed` command, `pull` command) and the corresponding rerank/generate ones with the helpers.

**Patch 2 — make the MCP server respect `--index`** (in `src/mcp/server.ts`):

- `startMcpServer` gains an optional parameter `opts?: { dbPath?: string }`. Replace `dbPath: getDefaultDbPath()` with `dbPath: opts?.dbPath ?? getDefaultDbPath()`.
- `startMcpHttpServer` receives the same treatment.

In `src/cli/qmd.ts`, inside `case "mcp"`:

- Replace `await startMcpServer()` with `await startMcpServer({ dbPath: getDbPath() })`.
- Replace `await startMcpHttpServer(port)` with `await startMcpHttpServer(port, { dbPath: getDbPath() })`.

**Patch 3 — allow hyphenated terms in semantic queries** (in `src/store.ts`):

- In `validateSemanticQuery()`, change the negation check so it only rejects `-term` when the hyphen starts a token:

```ts
if (/(^|\s)-(?=\S)/.test(query)) {
  return "Negation (-term) is not supported in vec/hyde queries. Use lex for exclusions.";
}
```

- Keep rejecting explicit negation such as `-multi-agent`, `performance -sports`, and `-"exact phrase"`.
- Accept natural hyphenated terms such as `built-in apps`, `rule for "built-in" apps`, `multi-agent memory`, `gpt-4 migration`, and `client-side routing`.
- Add focused coverage in `test/structured-search.test.ts` for both accepted hyphenated terms and rejected negation.

**Build follow-up observed while validating Patch 3**:

- `bun run build` may fail if the shared SQLite `Database` interface does not declare `transaction()`, even though migration code uses it.
- Fix by adding `transaction<T extends (...args: any[]) => any>(fn: T): T;` to the interface in `src/db.ts`.

### 2. Install locally

```sh
bun install
bunx tsc -p tsconfig.build.json --noCheck
printf '#!/usr/bin/env node\n' | cat - dist/cli/qmd.js > dist/cli/qmd.tmp \
  && mv dist/cli/qmd.tmp dist/cli/qmd.js && chmod +x dist/cli/qmd.js
bun link
bun link @tobilu/qmd
```

### 3. Configure the multilingual embedding variable

```sh
QMD_VAR='QMD_EMBED_MODEL="hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"'
case "$SHELL" in
  */zsh)  RC="$HOME/.zshrc"; LINE="export $QMD_VAR" ;;
  */bash) RC="$HOME/.bashrc"; [ ! -f "$RC" ] && RC="$HOME/.bash_profile"
          LINE="export $QMD_VAR" ;;
  */fish) RC="$HOME/.config/fish/config.fish"
          LINE='set -gx QMD_EMBED_MODEL "hf:Qwen/Qwen3-Embedding-0.6B-GGUF/Qwen3-Embedding-0.6B-Q8_0.gguf"' ;;
esac
grep -qF 'QMD_EMBED_MODEL' "$RC" || printf '\n# QMD multilingual embedding model\n%s\n' "$LINE" >> "$RC"
```

> **Important**: do NOT create an alias `qmd='bunx ...'` — it will override the patched binary.

### 4. Verify installation

```sh
which qmd                # should point to ~/.bun/bin/qmd, NOT a bunx alias
qmd --version
echo $QMD_EMBED_MODEL    # should be hf:Qwen/Qwen3-Embedding-0.6B-GGUF/...
```

## One-time setup (run from inside `wiki/`)

### 1. Collection

```sh
cd /Users/djalmajr/Developer/zommehq/buntime/wiki
qmd --index buntime collection add . --name wiki --mask "**/*.md"
```

One collection only. No `apps/`, `plugins/`, `charts/` from the repo — semantic search is restricted to the wiki.

### 2. Embeddings

```sh
qmd --index buntime embed   # downloads models on first run (~2GB); takes a few minutes
```

### 3. Hierarchical contexts

The contexts inject metadata into each result and encode two key rules:

1. **Separation by audience**: `apps/` is technical (runtime + plugins); `ops/` is operational; `data/` is schemas.
2. **Source-of-truth markers**: the wiki is canonical — source code outside `wiki/` is a reference, not a source.

```sh
# Global wiki context
qmd --index buntime context add / "Canonical wiki for the Buntime project — Bun runtime with worker pool and plugin system. This is the single source of truth for architecture, plugins, deploy, and operations. The source code repo is intentionally not indexed."

# Internal folders
qmd --index buntime context add qmd://wiki/apps "Technical documentation for the runtime, client apps (cpanel, cli, vault), shared packages, and the 10 core plugins. Audience: dev. Does NOT contain operational rules — those live in ops/."
qmd --index buntime context add qmd://wiki/ops "Operations: local dev, helm charts, release flow (GitHub + self-hosted GitLab), JSR publish, logging, performance, security. Audience: ops."
qmd --index buntime context add qmd://wiki/data "Stores and schemas: LibSQL via plugin-keyval and plugin-proxy, API key file store, in-memory caches, apps/plugins PVCs. Audience: dev."
qmd --index buntime context add qmd://wiki/sources "Summaries of ingest operations. Each file describes what was consolidated and which final pages were created/updated."
```

### 4. Verification

```sh
qmd --index buntime status                           # collection + doc count
qmd --index buntime context list                     # confirms attached contexts
qmd --index buntime query "how does TTL=0 work in the worker pool"
qmd --index buntime query "publish @buntime/shared to JSR"
qmd --index buntime query "difference between persistent and serverless plugin mode"
qmd --index buntime query "gateway rate limiting"
qmd --index buntime query "vec: rule for built-in apps" --no-rerank -n 3
qmd --index buntime query "vec: performance -sports" --no-rerank -n 1  # expected: rejected as explicit negation
```

## Local patch verification

After applying the local QMD patches in `~/Developer/github/qmd`, validate before relying on the MCP:

```sh
cd ~/Developer/github/qmd
bun test --preload ./src/test-preload.ts test/structured-search.test.ts
bun run build
cd /Users/djalmajr/Developer/zommehq/buntime
qmd --index buntime query "vec: rule for built-in apps" --no-rerank -n 3
qmd --index buntime query "vec: performance -sports" --no-rerank -n 1
```

Expected result:

- Focused structured-search tests pass.
- Build passes.
- The `built-in` query returns wiki results instead of failing validation.
- The `performance -sports` query still fails with `Negation (-term) is not supported in vec/hyde queries. Use lex for exclusions.`

Known caveat from the 2026-05-02 validation run: the full upstream QMD test suite can fail locally for environment/pre-existing reasons unrelated to the hyphen fix, including a `better-sqlite3` native module built for a different Node ABI, embedding dimension expectations of 768 while the current model returns 1024 dimensions, HTTP daemon test timeouts, and a native `ggml/Metal` assertion on process exit. Treat the focused structured-search test, build, CLI query smoke, and MCP smoke as the validation set for this patch.

## MCP — registered per project

The QMD MCP server is configured **per project**, not globally. Add a `.mcp.json` at the Buntime repo root for Claude-compatible MCP clients:

```json
{
  "mcpServers": {
    "qmd": {
      "command": "qmd",
      "args": ["--index", "buntime", "mcp"]
    }
  }
}
```

> **Note**: `command: "qmd"` points to the patched local binary at `~/.bun/bin/qmd` (installed via `bun link`). If you use `bunx @tobilu/qmd ... mcp`, the MCP would use the unpatched npm version — it generates query embeddings with the wrong model and vector search becomes unusable.

This ensures that in any Claude Code session opened inside this repo, the `qmd` MCP sees the Buntime wiki index — without crosstalk with other projects.

> Why **not** in the global `~/.claude.json`? Because a global MCP points to a single index — when you add other projects with their own indexes, you would be locked into one choice. Per-project, each repo chooses its own universe.

For Codex, mirror the same project-local registration in `.codex/config.toml`:

```toml
[mcp_servers.qmd]
command = "qmd"
args = ["--index", "buntime", "mcp"]
```

Do not add this server to the global `~/.codex/config.toml`; a global entry would make unrelated projects query the Buntime index. Verify from the repo root:

```sh
codex mcp list
codex mcp get qmd
```

## Keeping the index up to date

Whenever you edit wiki content:

```sh
qmd --index buntime update    # rescans the filesystem and re-indexes what changed
qmd --index buntime embed     # generates embeddings for new/modified chunks
```

Recommended alias — add to your shell rc:

```sh
alias buntime-refresh='qmd --index buntime update && qmd --index buntime embed'
```

The `/wiki-lint` skill reports when the index is stale (`needsEmbedding > 0` or divergence between the filesystem and `qmd --index buntime status`).

## MCP daemon (optional)

If you run multiple Claude Code sessions simultaneously, start QMD as an HTTP daemon to keep models in memory between sessions:

```sh
qmd --index buntime mcp --http --daemon              # localhost:8181
qmd --index buntime mcp stop                         # stop the daemon
```

Point MCP clients to `http://localhost:8181/mcp` (update `.mcp.json` to `"url": "http://localhost:8181/mcp"` instead of `command`/`args`). Advantage: GGUF models stay loaded in VRAM between sessions, eliminating cold starts for each new Claude Code window.
